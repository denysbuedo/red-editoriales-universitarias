Propuesta conceptual y tecnológica

Evolución del Portal de la Red Nacional de Editoriales Universitarias

De un directorio institucional a un catálogo nacional de libros, autores, colecciones y editoriales, preservando el concepto de Red como eje de identidad, articulación y unidad.

Ministerio de Educación Superior de Cuba
Documento de trabajo · Julio de 2026

1. Punto de partida

La propuesta preliminar recibida define correctamente un portal informativo para presentar la Red, publicar el directorio de editoriales miembros, explicar su estructura organizativa, difundir novedades, reunir enlaces de interés y facilitar el contacto.

Ese planteamiento constituye una base institucional válida. Sin embargo, presenta una limitación fundamental: describe con claridad quiénes integran la Red, pero apenas muestra qué producen las editoriales. La función esencial de cada editorial es editar, publicar, preservar, distribuir y promover libros y otras obras académicas.

Decisión propuesta: conservar la visibilidad institucional de la Red, pero colocar los libros y la producción editorial en el centro de la experiencia pública.

2. Concepto rector

El portal debe asumir una doble identidad, complementaria y no contradictoria:

La Red como institución

Da unidad, representación nacional, identidad compartida, cooperación, normas comunes, alianzas, noticias y visibilidad a cada editorial.

El catálogo como servicio público

Permite descubrir libros, autores, colecciones, temas y editoriales, y acceder a los sitios o plataformas donde las obras están disponibles.

La portada debe expresar ambas dimensiones con claridad: “Una Red, muchas voces editoriales” y “Libros que conectan universidades, autores y lectores”.

4. Arquitectura tecnológica

El portal no debe convertirse en un tercer sistema de gestión. Debe actuar como una capa pública de presentación, búsqueda, integración y posicionamiento.

Sistema de gestión de editorialesEditoriales, responsables, contactos, universidades, provincias y estados
→
Portal RNEUExperiencia pública, navegación, buscador, SEO y páginas institucionales
←
Omeka SLibros, autores, colecciones, metadatos, portadas, ficheros y relaciones
Principio de integración: cada dato debe tener una única fuente maestra. El portal consulta las APIs, transforma los datos para presentación y mantiene solamente caché e índices de búsqueda cuando sea necesario.

4.1 Sistema de gestión de editoriales

El sistema existente debe seguir siendo la fuente oficial de:

  • nombre y sigla de la editorial;
  • universidad o institución de adscripción;
  • provincia y ubicación;
  • director, responsable de ISBN y contactos;
  • logotipo y sitio web;
  • estado de membresía y fecha de actualización.

Se recomienda exponer una API REST versionada, por ejemplo /api/v1/editoriales, con identificadores estables, paginación, filtros, fecha de modificación y documentación OpenAPI.

4.2 Gestión de libros en Omeka S

Omeka S es una opción adecuada para este proyecto porque está orientado a instituciones que necesitan gestionar un conjunto compartido de recursos y publicarlo en uno o varios sitios. Permite trabajar con elementos, conjuntos, vocabularios, plantillas de recursos, relaciones entre recursos y una API propia.

Se propone utilizar Omeka S como repositorio descriptivo y de publicación de:

  • libros y capítulos;
  • autores y colaboradores;
  • colecciones editoriales;
  • editoriales como entidades enlazables;
  • portadas, PDF, EPUB y otros medios cuando corresponda;
  • enlaces a repositorios institucionales, eLibro u otras plataformas.

LibroTítulo, subtítulo, autores, ISBN, fecha, editorial, colección, edición, idioma, resumen, palabras clave, materia, licencia, formato, portada, URL de acceso y disponibilidad
AutorNombre normalizado, afiliación, ORCID, biografía y publicaciones relacionadas
EditorialIdentificador del sistema maestro, nombre, universidad, logo, provincia y URL institucional
ColecciónNombre, descripción, editorial responsable, ISSN si aplica y libros asociados

Las plantillas de Omeka S deben basarse en vocabularios normalizados, combinando Dublin Core Terms, Bibliographic Ontology y propiedades específicas estrictamente necesarias.

6. Criterio sobre Omeka

Por qué sí
  • software libre y orientado a colecciones;
  • metadatos estructurados;
  • relaciones entre libros, autores y colecciones;
  • API REST;
  • varios sitios sobre una misma instalación;
  • integración con IIIF y visores;
  • importación masiva.
Precauciones
  • no sustituye un sistema editorial de flujo de trabajo;
  • requiere definir previamente el modelo de metadatos;
  • la calidad del catálogo depende de normalización y control de autoridades;
  • debe evaluarse almacenamiento de PDF y derechos de acceso;
  • el buscador público puede requerir un índice adicional al crecer.
Recomendación: utilizar Omeka S, no Omeka Classic, porque el proyecto es institucional, multi-editorial, relacional y potencialmente multisede.

7. Flujo de publicación recomendado

  1. La editorial actualiza sus datos institucionales en el sistema de gestión existente.
  2. El responsable registra o importa el libro en Omeka S mediante una plantilla normalizada.
  3. Se valida ISBN, autorías, editorial, colección, licencia y URL de acceso.
  4. Omeka S publica el registro y expone sus datos mediante API.
  5. El portal RNEU sincroniza o consulta el registro y crea la página pública optimizada para buscadores.
  6. El usuario descubre el libro y accede al PDF, EPUB, repositorio, eLibro o portal editorial según corresponda.

8. SEO y visibilidad

Cada libro, autor, colección y editorial debe tener una URL estable. Las fichas de libros deben incorporar datos estructurados de Schema.org, especialmente Book, Person, Organization y BreadcrumbList.

Ejemplos de URL:

  • /libros/inteligencia-artificial-aplicada/
  • /autores/maria-perez-garcia/
  • /editoriales/editorial-feijoo/
  • /colecciones/ciencias-de-la-educacion/

El portal debe producir sitemap XML, metadatos sociales, títulos únicos, canonicales, páginas rápidas y contenido HTML rastreable sin depender completamente de JavaScript.

9. Hoja de ruta

Fase 1

Validar nombre, arquitectura, fuentes maestras y modelo de datos.

Fase 2

Publicar directorio desde la API del sistema existente y desplegar Omeka S.

Fase 3

Importar catálogo inicial, normalizar autores, editoriales y colecciones.

Fase 4

Activar buscador nacional, analítica, SEO, indicadores y mejoras continuas.

10. Conclusión

La propuesta inicial debe mantenerse como fundamento institucional, pero ampliarse con una visión editorial y de servicio público. La Red debe verse y sentirse como una organización unida; los libros deben demostrar el valor concreto de esa unidad.

El resultado esperado no es solamente un portal corporativo ni un repositorio aislado. Es un ecosistema nacional de descubrimiento editorial universitario, integrado mediante APIs, con Omeka S como plataforma de colecciones y el portal RNEU como experiencia pública común.
